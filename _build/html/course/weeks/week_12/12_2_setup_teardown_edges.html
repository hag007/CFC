12.9. עבודה נכונה עם setUp, תרחישים ושוליים#

במחברת זו נעמיק בשיטות עבודה טובות בבדיקות יחידה עם unittest.
נלמד כיצד להכין סביבה לבדיקה (setup), לנקות אותה (teardown), לבדוק תרחישים שונים באותה פונקציה,
ולהשתמש ב־subTest כדי לבחון סדרות של נתונים בקלות.

12.9.1. מטרות הלמידה#

  • להבין את תפקידן של הפונקציות ()setUp ו־()tearDown.

  • לתכנן בדיקות לפי דפוס AAA (Arrange → Act → Assert).

  • ללמוד להשתמש ב־subTest לבדיקות חוזרות על נתונים שונים.

  • להוסיף בדיקות לקלטים בקצוות (edge cases) ולוודא טיפול נכון בשגיאות.

12.9.2. setUp ו־tearDown#

במקרים רבים כמה בדיקות משתמשות באותם נתוני התחלה או בהגדרות משותפות.
במקום לשכפל את הקוד הזה בכל פונקציה, נוכל להשתמש ב־()setUp.

  • ()setUp — רצה לפני כל בדיקה (כל מתודת test_).

  • ()tearDown — רצה אחרי כל בדיקה (גם אם היא נכשלה).
    נשתמש בה בעיקר לניקוי משאבים (למשל סגירת קובץ, מחיקת משתנה זמני).

12.9.2.1. דוגמה בסיסית:#

  • setUp()
    רצה לפני כל בדיקה (test) ומגדירה משתנים משותפים — במקרה זה:
    self.length, self.width ו־self.area.
    כך כל בדיקה מתחילה מאותו מצב בסיסי, בלי צורך להגדיר את הערכים מחדש בכל פונקציה.

  • ()tearDown
    רצה אחרי כל בדיקה, גם אם הבדיקה נכשלה.
    כאן משתמשים בה לניקוי המשתנים (השמה של None) או לצורכי ניפוי שגיאות (debugging).

  • ()test_area_positive
    בודקת שהשטח המחושב (length * width) תואם לערך הצפוי שנשמר ב־self.area.

  • ()test_perimeter
    בודקת את חישוב ההיקף לפי הנוסחה ( P = 2(l + w) ),
    ומוודאת שהתוצאה שווה ל־16 במקרה הנתון (אורך 5 ורוחב 3).

12.9.3. דפוס Arrange–Act–Assert (AAA)#

זהו סגנון כתיבה שמסייע לשמור על קריאות הבדיקה:

  1. Arrange – הגדרת הנתונים והאובייקטים.

  2. Act – הפעלת הפעולה שאנו בודקים.

  3. Assert – בדיקה שהתוצאה תואמת לציפיות.

12.9.3.1. דוגמה:#

בדוגמה זו אנו מדגימים כיצד כותבים בדיקות יחידה בצורה קריאה וברורה בעזרת דפוס AAA —
שלושת השלבים הקלאסיים של כל בדיקה:

  1. Arrange (הכנה)
    בשלב זה אנחנו מגדירים את נתוני הקלט ואת התוצאה הצפויה — במקרה הזה, בסיס (base) וגובה (height) של משולש,
    ואת הערך הצפוי של שטחו לפי הנוסחה ( A = \frac{1}{2} \times base \times height ).

  2. Act (ביצוע)
    כאן אנחנו מפעילים את הפונקציה הנבדקת (triangle_area) עם הנתונים שהכנו,
    ושומרים את התוצאה במשתנה result.

  3. Assert (בדיקה)
    לבסוף, אנחנו משתמשים בפקודת assertEqual כדי לבדוק שהתוצאה שחושבה
    אכן תואמת לערך הצפוי (expected).

הבדיקות כוללות שני מקרים:

  • בדיקה רגילה של משולש תקין (base=10, height=5 → שטח 25).

  • בדיקה של מקרה גבול שבו הבסיס הוא אפס (שטח צפוי 0).

Note

ניתן להבין במבט אחד מה נבדק, איך זה נבדק, ומה התוצאה המצופה.

12.9.4. שימוש ב־subTest לתרחישים מרובים#

לעיתים נרצה לבדוק את אותה פונקציה עבור מגוון נתונים שונים.
במקום לכתוב בדיקה נפרדת לכל מקרה, נוכל להשתמש ב־()subTest.

12.9.4.1. דוגמה:#

בדוגמה זו אנו בודקים את הפונקציה square(x) שמחזירה את הערך ( x^2 ).
במקום לכתוב בדיקה נפרדת לכל ערך קלט, אנחנו משתמשים במנגנון ()subTest של unittest.

כך זה עובד:

  • אנו יוצרים רשימה של זוגות (input, expected) — כל אחד מייצג מקרה בדיקה קטן.

  • בתוך לולאה, עבור כל זוג, נשתמש ב־with self.subTest(x=inp): כדי ליצור תת־בדיקה נפרדת.

  • בכל subTest, נבדוק שהתוצאה של square(inp) קרובה לערך הצפוי בעזרת assertAlmostEqual
    (מתאים במיוחד למספרים עשרוניים).

היתרון:
אם אחת מהתת־בדיקות נכשלת, הדוח של unittest מציין בדיוק איזה ערך קלט גרם לכשל —
כך קל מאוד לזהות שגיאות ספציפיות בלי לעצור את שאר הבדיקות.

12.9.5. בדיקות שוליים (Edge Cases)#

בדיקות טובות לא מסתפקות בערכים “רגילים” —
צריך לוודא גם שהקוד מתמודד נכון עם קלטים חריגים: אפס, שלילי, None, טיפוסים שגויים וכו’.

נניח פונקציה שמחשבת שטח מלבן, אבל אוסרת על ערכים שליליים:

12.9.5.1. דוגמה: בדיקות שוליים (Edge Cases)#

  1. test_negative_values_raise

def rectangle_area(length, width):
    if length < 0 or width < 0:
        raise ValueError("Length and width must be non-negative")
    return length * width

        with self.assertRaises(ValueError):
            rectangle_area(-1, 2)
        with self.assertRaises(ValueError):
            rectangle_area(3, -5)

    def test_zero_dimension(self):
        """Zero length or width should return 0."""
        self.assertEqual(rectangle_area(0, 5), 0)

def circle_area(radius):
    """Compute the area of a circle with non-negative radius."""
    if radius < 0:
        raise ValueError("Radius must be non-negative")
    return math.pi * radius ** 2


class TestCircleArea(unittest.TestCase):
    def setUp(self):
        """Prepare test data before each test."""
        # List of (radius, expected_area) tuples
        self.test_cases = [
            (0, 0.0),
            (1, math.pi),
            (2, 4 * math.pi),
        ]

    def test_circle_area_values(self):
        """Check multiple radius values using subTest."""
        for radius, expected in self.test_cases:
            with self.subTest(radius=radius):
                result = circle_area(radius)
                self.assertAlmostEqual(result, expected, places=6)

    def test_negative_radius_raises(self):
        """Verify that negative radius raises ValueError."""
        with self.assertRaises(ValueError):
            circle_area(-1)

12.9.7. סיכום#

במחברת זו למדנו:

  • כיצד להשתמש ב־()setUp ו־()tearDown כדי להכין סביבה לבדיקה.

  • כיצד לארגן בדיקות לפי דפוס AAA להגברת קריאות.

  • כיצד לכתוב תרחישים חוזרים בעזרת subTest.

  • כיצד לטפל במקרי קצה (Edge Cases) ולהבטיח שהקוד עמיד גם בתנאים לא צפויים.

במחברת הבאה נמשיך לבדוק מחלקות ומורשות (OOP Testing) ונכיר את השימוש הבסיסי ב־unittest.mock.
